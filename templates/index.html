Recommendations

1

You type your password

Your password stays on your device the whole time. Nothing is sent in plain text, not to us, not to anyone. You could disconnect from the internet right now and the strength check would still run.

2

It gets turned into a fingerprint

Your password is converted into a unique scrambled code called a hash. This is a one-way process: nobody can reverse that code back to your original password.

3

Only a tiny piece is sent out

Imagine that fingerprint is 40 characters long. We only send the first 5 to HaveIBeenPwned. That small piece is enough to search their database without ever exposing which specific password is yours.

4

A list comes back

HaveIBeenPwned responds with hundreds of leaked codes that start with those same 5 characters. Your device compares privately and locally. Nobody learns which password you were checking.

5

You get your result

If your password matches one in the leaked list, we show you how many times it appeared in real breaches. No match means you are in the clear in our dataset, though no tool can guarantee absolute safety online.

1
SHA-1 hashing (client-side)
Your password is hashed using SHA-1. For example: "password" becomes 5BAA61E4C9B93F3F0682250B6CF8331B7EE68FD8. This is deterministic and one-way: the original input cannot be recovered from the hash.
2
k-anonymity prefix query
Only the first 5 hex characters (5BAA6) are sent to the HIBP Range API: GET api.pwnedpasswords.com/range/5BAA6. The server never knows which specific password is being queried. This is the k-anonymity model, originally proposed by Cloudflare and HIBP in 2018.
3
Padded range response
The API returns roughly 800 hash suffixes (the 35 characters after the prefix) along with a breach count for each. The Add-Padding: true request header pads the response to a constant size, which neutralises traffic-analysis attacks.
4
Local suffix comparison
The remaining 35 characters of your hash are compared against the returned list entirely on your device. No additional network request is made. Zero knowledge of your input is ever transmitted.
5
Breach count interpretation
A matching suffix returns a count representing how many times that exact password appeared in HIBP's corpus of 13+ billion records. A count of 0 means no match, which is a strong positive signal, though not an absolute safety guarantee.

About HaveIBeenPwned

Created by security researcher Troy Hunt in 2013, HaveIBeenPwned aggregates data from hundreds of public data breaches into a searchable database of over 13 billion compromised accounts. The Pwned Passwords service catalogues 847+ million unique password hashes. The k-anonymity Range API was co-designed with Cloudflare and published as an open, auditable protocol.

Common misconceptions

Things people get wrong

01

Letter-to-symbol swaps add very little real security. Replacing "E" with "3" or "a" with "@" is one of the first substitution patterns password-cracking software tries. It barely increases the time to crack.

Source: NIST SP 800-63B: Digital Identity Guidelines

02

8 characters is no longer a safe minimum. A modern GPU can exhaust all possible 8-character passwords in minutes. The current guidance is at least 14 characters.

Source: Hive Systems Password Cracking Table, 2023

03

A strong password is not safe if you reuse it. If one website stores passwords in plaintext and gets breached, every account sharing that password is instantly compromised regardless of its complexity.

Source: Verizon 2022 DBIR: Credential Stuffing Attacks

04

Passphrases beat short complex passwords. "correct-horse-battery-staple" at 29 characters is far stronger than "P@ssw0rd!" at 9 characters and significantly easier to remember. Length beats complexity every time.

Source: XKCD #936: Password Strength & NIST SP 800-63B

05

Frequent mandatory password changes can backfire. Forced resets lead users to pick weaker incremental variations. NIST now recommends only changing passwords when there is evidence of compromise.

Source: NIST SP 800-63B §5.1.1: Memorized Secret Authenticators
